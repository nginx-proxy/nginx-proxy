# Troubleshooting

## nginx‑proxy container exits with "hello from palo alto networks"

Some users have observed that the **nginx‑proxy** container terminates shortly after it starts and the last line in the logs looks like:
```
hello from palo alto networks
```
This message is **not** an error generated by *nginx‑proxy* itself. It is produced by a **Docker health‑check** that is defined in the official `nginx‑proxy` image. The health‑check attempts to contact the container on port 80; the default response from the health‑check script is the literal string *"hello from palo alto networks"*. When the health‑check fails repeatedly Docker marks the container as unhealthy and, depending on the orchestration tool (e.g., Docker‑Compose with `restart: on-failure`), the container may be stopped.

### Why does it happen?
- The health‑check is a leftover from an older upstream image that used a Palo Alto Networks‑provided health‑check binary.
- If the container does not expose a plain HTTP endpoint that returns a 2xx status code, the health‑check will fail.
- Using the `latest` tag can pull a newer image where the health‑check behaviour changed, making the issue more visible.

### How to resolve / work around the issue
1. **Disable the built‑in health‑check**
   ```yaml
   services:
     nginx-proxy:
       image: nginxproxy/nginx-proxy:1.10
       healthcheck: {disable: true}
   ```
   or, in a `docker run` command:
   ```bash
   docker run --detach \
       --name nginx-proxy \
       --publish 80:80 \
       --volume /var/run/docker.sock:/tmp/docker.sock:ro \
       --health-cmd "exit 0" \
       nginxproxy/nginx-proxy:1.10
   ```
2. **Provide a simple HTTP endpoint** that returns `200 OK` on `/` (or the path used by the health‑check). Most applications already do this, but if you rely solely on `VIRTUAL_HOST` without exposing a root path, add a tiny web server that answers `200`.
3. **Pin the image version** instead of using `latest`. The issue is documented for versions prior to `1.10`. Using a stable tag (e.g., `nginxproxy/nginx-proxy:1.10`) avoids unexpected changes.
4. **Upgrade to the latest stable release** where the health‑check has been removed. Check the [release notes](https://github.com/nginx-proxy/nginx-proxy/releases) for a version that no longer includes the Palo Alto health‑check.

### Additional notes
- The log line itself is harmless; the container does not crash because of the message.
- If you need the health‑check for monitoring, replace it with a custom one that matches your environment.
- For more detailed debugging, run the container with `docker logs --tail 100 <container>` to see the health‑check attempts.
